# 模拟实现 UNIX 文件系统 —— 说明文档

---

## 一、项目概述

本项目实现了一个模拟 UNIX 文件系统的完整系统，命名为 UPFS。在宿主操作系统上通过单个磁盘镜像文件模拟虚拟磁盘设备，在其上构建了超级块、索引节点区和数据区三段式文件卷结构，采用成组链接法管理空闲磁盘块，采用混合索引结构组织文件物理块，实现了多用户管理和多级目录体系。此外，还附带了操作系统内核层（进程管理、内存管理、虚拟机执行引擎、时间片轮转调度、系统调用接口），以更完整地呈现操作系统各子系统间的协作关系。

编程语言为 C（C17 标准），仅依赖标准 C 库和 POSIX 线程库，可在 Linux 和 macOS 上编译运行。

## 二、环境配置与编译运行

### 2.1 编译环境要求

| 项目 | 要求 |
|------|------|
| 操作系统 | Linux 或 macOS（需 POSIX 兼容） |
| 编译器 | GCC 或 Clang（支持 C17） |
| 构建工具 | CMake 3.10 及以上版本 |
| 线程库 | pthreads（一般系统自带） |

无需安装任何第三方库。

### 2.2 编译步骤

```
cmake -B build -S . -DCMAKE_C_FLAGS="-D_GNU_SOURCE"
cmake --build build
```

编译完成后，可执行文件位于 `build/bin/OS_design`。

### 2.3 运行方式

**交互式终端模式**（推荐验收使用）：

```
./build/bin/OS_design
```

程序启动后会自动扫描当前目录及上级目录中的磁盘镜像文件。若首次运行，需输入 `format` 命令创建文件卷；之后可通过 `mount` 命令恢复已有数据。

**多终端服务模式**：

```
./build/bin/OS_design --serve        # 默认端口 4096
./build/bin/OS_design --serve 9999   # 自定义端口
```

该模式下，可通过浏览器访问 `http://localhost:8080` 的网页终端，或通过 `nc localhost 4096` 连接原始终端，支持多个终端同时操作同一文件系统。

## 三、文件卷组织结构

### 3.1 磁盘布局

虚拟磁盘以块为基本单位进行管理，各区域划分如下：

| 区域 | 块号范围 | 块数 | 说明 |
|------|----------|------|------|
| 引导块 | 0 号 | 1 | 存放引导标识 |
| 超级块 | 1 号 | 1 | 文件系统全局管理信息 |
| 索引节点区 | 2～33 号 | 32 | 存放所有磁盘索引节点 |
| 数据区 | 34～545 号 | 512 | 目录数据块和文件数据块 |

核心参数设定：

| 参数名 | 值 | 说明 |
|--------|-----|------|
| 块大小 | 512 字节 | 参考课本设定 |
| 磁盘索引节点大小 | 32 字节 | 每块可容纳 16 个 |
| 索引节点总数 | 512 个 | 32 块 × 16 个/块 |
| 目录项大小 | 16 字节 | 文件名 14 字节 + 节点号 2 字节 |
| 每块目录项数 | 32 个 | 512 / 16 |
| 空闲块栈容量 | 50 | 成组链接法每组最多 50 块 |
| 索引地址数 | 10 个 | 8 直接 + 1 一级间址 + 1 二级间址 |
| 每用户最大打开文件数 | 20 | 用户打开文件表容量 |
| 系统打开文件表容量 | 40 | 全局共享表 |
| 最大用户数 | 10 | |

### 3.2 超级块结构

超级块存放在磁盘 1 号块，恰好占满 512 字节，包含以下关键字段：

- **魔数**：用于挂载时校验文件系统有效性
- **索引节点统计**：总数、空闲数
- **数据块统计**：总数、空闲数
- **空闲块栈**：当前组的空闲块号数组（最多 50 个）及有效条目数
- **空闲块链指针**：指向下一组登记块的块号
- **空闲索引节点栈**：缓存的空闲节点号及栈顶指针

### 3.3 成组链接法

空闲数据块采用成组链接法管理，工作原理如下：

**分配 `balloc()`**：从超级块中空闲块栈的栈顶弹出一个块号。若栈空，则读取登记块链中下一个登记块，将其记录的一组块号装入栈中，继续弹出。

**回收 `bfree()`**：将释放的块号压入栈顶。若栈已满（已有 50 个），则将当前栈中所有块号写入被释放块（该块成为新的登记块），清空超级块栈，仅保留该登记块自身。

### 3.4 磁盘索引节点

每个文件或目录对应一个 32 字节的磁盘索引节点，包含：

| 字段 | 类型 | 说明 |
|------|------|------|
| 文件类型与权限 | 2 字节 | 目录/普通文件标识，读写执行权限位 |
| 硬链接计数 | 2 字节 | 指向该节点的目录项数量 |
| 文件长度 | 4 字节 | 以字节为单位 |
| 所有者用户号 | 2 字节 | |
| 所有者组号 | 2 字节 | |
| 直接索引 | 8 × 2 字节 | 直接指向 8 个数据块，覆盖 4 千字节 |
| 一级间址 | 2 字节 | 指向一个索引块，该块存放 256 个块号，覆盖 128 千字节 |
| 二级间址 | 2 字节 | 两层索引，最大可覆盖约 32 兆字节 |

索引节点的分配（`ialloc`）和释放（`ifree`）采用栈式管理，超级块中缓存一批空闲节点号，栈空时扫描磁盘索引节点区补充。

### 3.5 内存索引节点与哈希缓存

磁盘索引节点被读入内存后，封装为内存索引节点，增加以下管理信息：

- 磁盘节点号
- 引用计数
- 脏标志（是否需要回写磁盘）
- 读写锁（支持并发访问）

内存中维护 128 个哈希桶，以节点号取模为键，链地址法解决冲突。`iget()` 获取节点时优先查缓存，命中则增加引用计数；`iput()` 释放节点时递减计数，归零后若为脏则回写磁盘，若链接计数为零则回收该节点。

### 3.6 两层打开文件表

本系统实现了课本所述的两层打开文件表结构：

**用户打开文件表**：每个用户持有一张，最多 20 个表项，每项记录文件读写位置、打开模式、关联的内存索引节点。用户通过文件描述符（0～19 的整数）访问。

**系统打开文件表**：全局唯一，最多 40 个表项。每当用户打开文件时，同时在系统表中分配一个表项，记录文件偏移量、打开模式、引用计数和索引节点指针。多个用户打开同一文件时共享同一系统表项，引用计数随之增减。关闭文件时递减引用计数，归零则释放表项。

### 3.7 目录结构

目录是一种特殊文件，其数据内容由定长 16 字节的目录项组成。每个目录项包含 2 字节的索引节点号和 14 字节的文件名。节点号为零表示该目录项空闲。

每个目录至少包含 `.`（指向自身）和 `..`（指向父目录）两个条目。

### 4.5 权限控制

每个文件和目录的索引节点中存储了 9 位权限位（所有者读/写/执行、同组读/写/执行、其他人读/写/执行），遵循 UNIX 的三级权限模型。

`vfs_access()` 函数根据当前用户的身份（用户号和组号），按照所有者、同组、其他人的优先级依次匹配权限位，判断是否具有指定的访问权限。超级用户不受权限限制。

## 六、格式化所做的工作

执行 `format` 命令时，系统依次完成以下步骤：

1. 在内存中分配一块与虚拟磁盘等大的空间
2. 写入引导块（0 号块），标记引导标识
3. 清零整个索引节点区（32 个块）
4. 统计数据区空闲块（共 511 块，根目录占用 1 块），按成组链接法组织空闲块链
5. 初始化空闲索引节点栈
6. 写入超级块（1 号块）
7. 创建根目录（索引节点 1 号），写入 `.` 和 `..` 两个目录项
8. 将内存中的磁盘镜像保存到宿主机文件
9. 自动挂载刚创建的文件系统
10. 创建系统目录结构（`/bin`、`/etc`、`/home`、`/root`、`/tmp`、`/usr` 等）
11. 初始化用户数据库
12. 将预置的演示程序注入 `/bin` 目录
13. 设置系统环境变量（`PATH=/bin:/usr/bin`）
14. 提示创建首个用户账户

## 七、影响可管理文件长度的因素

1. **直接索引数量**：8 个直接索引可寻址 8 × 512 = 4096 字节
2. **间接索引级数**：一级间址扩展 256 × 512 = 131072 字节，二级间址扩展 256 × 256 × 512 ≈ 33554432 字节
3. **块大小**：块越大，每级索引能覆盖的范围越大
4. **块号宽度**：本系统使用 16 位块号，最多寻址 65536 个块
5. **数据区块数**：本系统数据区仅有 512 块，实际可存放文件总量约 256 千字节
6. **索引节点中地址项数目**：本系统共 10 个地址项（8 直接 + 1 一级间址 + 1 二级间址）
7. **文件长度字段宽度**：使用 32 位无符号整数记录文件字节数，理论上限为 4 吉字节

```
testuser:~ > cp /data/test.txt /data/copy.txt
  File copied
```

### 8.5 硬链接

创建硬链接后，两个文件名指向同一个索引节点，链接计数变为 2：

```
testuser:~ > ln /data/test.txt /data/link.txt
  Hard link created

testuser:~ > stat /data/test.txt

  File:     /data/test.txt
  Inode:    347
  Type:     regular file
  Mode:     000644
  Links:    2
  Size:     11 bytes
  Owner:    uid=1000 gid=1000

testuser:~ > stat /data/link.txt

  File:     /data/link.txt
  Inode:    347
  Type:     regular file
  Mode:     000644
  Links:    2
  Size:     11 bytes
  Owner:    uid=1000 gid=1000
```

可以看到两个文件的索引节点号均为 347，链接计数均为 2，证实它们共享同一物理存储。

### 8.6 权限管理

使用 `chmod` 修改文件权限：

```
testuser:~ > chmod /data/test.txt 0755
  Mode changed

testuser:~ > stat /data/test.txt

  File:     /data/test.txt
  Inode:    347
  Type:     regular file
  Mode:     000755
  Links:    2
  Size:     11 bytes
  Owner:    uid=1000 gid=1000
```

权限从 644（所有者读写，其他人只读）变更为 755（所有者读写执行，其他人读和执行）。

其中 `d` 表示目录，`-` 表示普通文件。注意 `test.txt` 和 `link.txt` 的节点号相同（347），因为它们是硬链接关系。

## 九、主要数据结构间的关系总结

系统运行时涉及的核心数据结构及其关联关系如下：

1. **用户结构**持有用户标识、当前目录节点号和用户打开文件表
2. **用户打开文件表**中每个表项通过指针指向**系统打开文件表**中的对应表项
3. **系统打开文件表**中每个表项记录文件偏移量和引用计数，并通过指针指向**内存索引节点**
4. **内存索引节点**通过哈希表组织缓存，包含磁盘索引节点的副本，并在引用计数归零且为脏时回写磁盘
5. **磁盘索引节点**中的直接索引和间接索引块号指向**数据区**中的实际数据块
6. **超级块**中的空闲块栈和空闲节点栈分别管理数据块和索引节点的分配回收
7. **目录文件**的数据块中存放目录项数组，每个目录项通过节点号关联到对应的索引节点

以上各层结构自顶向下形成了从用户操作到物理存储的完整映射链路。
