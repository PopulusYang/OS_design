## 一、本部分在做什么

### 1.1 一句话目标

在普通文件（`testimg/vfs_disk.img`）里**模拟一块 273KB 的硬盘**，并只提供一种能力：**按「块号」读写，每块固定 512 字节**。

### 1.2 在整体项目中的位置

```
Shell 命令 format / mount / ls / cat …
        ↓
VFS、目录、inode、缓存、日志 …（后续章节）
        ↓
★ disk_io：disk_read_block / disk_write_block ★  ← 本部分
        ↓
宿主文件 vfs_disk.img（或内存中的 g_disk_mem）
```

**机理**：真实操作系统里，文件系统不直接操作「字节流文件」，而是对**块设备驱动**发读写请求；UPFS 用 `disk_io.c` 扮演这个最底层驱动。

### 1.3 本部分「做」与「不做」

| 本部分做 | 本部分不做 |
|----------|------------|
| 定义盘有多大、块多大 | 解析路径 `/home/a.txt` |
| 创建全零空盘、加载镜像 | 维护目录项、inode 表 |
| 按块号 memcpy 读写 | 块缓存 LRU（第二部分） |
| 把内存盘保存到 `.img` | 进程、调度、Shell 命令解析 |

## 二、内在原理（先建立图景）

### 2.1 为什么按「块」而不是按「字节」

硬盘、SSD 控制器 historically 以 **sector（扇区）** 为最小传输单位，常见 512B。文件系统因此：

- 用**块号** `block_no` 寻址；
- 一次读写**整块**（本项目中 `BLOCK_SIZE = 512`）。

好处：对齐、简化元数据（「一个 inode 占 32B，一块能放 16 个」）、便于缓存（后面 `buf.c` 以块为单位）。

### 2.2 虚拟磁盘 = 一大段连续字节

逻辑上把盘看成数组：

```
字节偏移 = block_no × 512
g_disk_mem[offset .. offset+511]  ← 第 block_no 块
```

对应代码中的核心公式（读写两处相同）：

```203:217:src/fs/disk_io.c
    offset = (size_t)block_no * (size_t)BLOCK_SIZE;
    memcpy(buf, g_disk_mem + offset, (size_t)BLOCK_SIZE);
    // ...
    offset = (size_t)block_no * (size_t)BLOCK_SIZE;
    memcpy(g_disk_mem + offset, buf, (size_t)BLOCK_SIZE);
```

### 2.3 磁盘布局（本部分只需记住块号）

常量定义在 `vfs_core.h`：

```10:39:include/vfs_core.h
#define BLOCK_SIZE              512     // 磁盘块大小
#define BG_COUNT                8       // 块组数
#define BG_BLOCKS_PER_GROUP     64      // 块组内块数
#define BG_ZONE_START           2       // 块组区起始块号
#define JOURNAL_ZONE_BLOCKS     32      // 日志区块数
#define TOTAL_DISK_BLOCKS       (2 + BG_TOTAL_BLOCKS + JOURNAL_ZONE_BLOCKS)
#define BOOT_BLOCKNO            0
#define SUPERBLOCK_BLOCKNO      1
#define ROOT_DIR_BLOCK          (BG_ZONE_START + 2)
#define VFS_MAGIC               0x55504653U
```

**计算**：`8 × 64 + 2 + 32 = 546` 块 → `546 × 512 = 279552` 字节 ≈ **273 KB**。

```
块号:  0          1              2 ────────────── 513    514 ─── 545
      ┌────┐    ┌────┐          ┌────────────────┐    ┌─────────┐
      │引导│    │超级│          │   8 个块组      │    │  日志区  │
      └────┘    └────┘          └────────────────┘    └─────────┘
```

格式化时，**第一步**就会往块 0 写引导魔数（见 `format.c` 的 `init_boot_block`），仍通过 `disk_write_block(0, …)` 完成——证明上层全部依赖本层接口。

### 2.4 两种「盘」的生命周期

| 阶段 | 函数 | 内存形态 | 持久化 |
|------|------|----------|--------|
| 新建空盘 | `disk_create()` | `calloc` 全零 | 尚未写文件 |
| 格式化结束 | `disk_save(path)` | 同上 | 写出 `vfs_disk.img` |
| 再次启动 mount | `disk_load(path)` | `mmap` 映射文件 | 改内存即改文件视图 |
| 卸载/退出 | `disk_shutdown()` | 释放 / `munmap` | `msync` 刷盘 |

**机理**：`mmap(MAP_SHARED)` 让「内存指针 `g_disk_mem`」与「磁盘文件」共享同一物理页；`disk_write_block` 改的是内存，真正落盘靠 `msync` / `disk_sync`。

## 三、代码结构讲解

### 3.1 对外 API（`disk_io.h`）

```13:40:src/fs/disk_io.h

```

演示时记住三个即可：**create（空盘）→ read/write（块）→ load/save（文件）**。

### 3.2 全局状态：谁持有「整块盘」

```17:27:src/fs/disk_io.c
static uint8_t *g_disk_mem = NULL;
static int    g_disk_fd   = -1;
static size_t g_disk_size = 0;
static char g_disk_path[512];
static int g_disk_dirty = 0;
```

- 全项目只有这里持有「盘」的基地址；
- 其它模块禁止自己 `malloc` 一块假盘，必须通过 `disk_read_block` / `disk_write_block` 访问。

### 3.3 边界检查：防止块号越界

```30:42:src/fs/disk_io.c
static int disk_check_block(int block_no, const void *buf)
{
    if (g_disk_mem == NULL) return -1;
    if (buf == NULL) return -1;
    if (block_no < 0 || block_no >= TOTAL_DISK_BLOCKS) return -1;
    return 0;
}
```

**机理**：文件系统 bug 若传入 `block_no = 9999`，会写穿内存；此处是最后一道防线。

### 3.4 创建空盘：`disk_create`

```44:63:src/fs/disk_io.c
int disk_create(void)
{
    // 若已有旧映射，先 munmap/close/free
    g_disk_mem = (uint8_t *)calloc((size_t)TOTAL_DISK_BLOCKS, (size_t)BLOCK_SIZE);
    if (g_disk_mem == NULL) return -1;
    g_disk_dirty = 0;
    g_disk_path[0] = '\0';
    g_disk_fd = -1;
    return 0;
}
```

`format` 的第一句磁盘相关操作：

```63:66:src/fs/format.c
    if (disk_create() != 0) {
        FORMAT_LOG("错误：内存分配失败");
        return -1;
    }
```

### 3.5 写入引导块：上层如何用块接口

```17:29:src/fs/format.c
static int init_boot_block(void)
{
    char boot_buf[BLOCK_SIZE];
    memset(boot_buf, 0, sizeof(boot_buf));
    memcpy(boot_buf, BOOT_MAGIC, sizeof(BOOT_MAGIC) - 1);  // "UPFSBOOT"
    // 块尾写入超级块号、根目录块号
    return disk_write_block(BOOT_BLOCKNO, boot_buf);
}
```

**演示亮点**：块 0 的前几个字节应是 ASCII `UPFSBOOT`，可用 `hexdump` 在宿主机直接验证（见第四节）。

### 3.6 加载镜像：`disk_load` 与严格大小校验

```74:87:src/fs/disk_io.c
    expect = (size_t)TOTAL_DISK_BLOCKS * (size_t)BLOCK_SIZE;
    fd = open(disk_path, O_RDWR);
    if (fstat(fd, &st) != 0 || (size_t)st.st_size != expect) {
        fprintf(stderr, "[disk_io] fstat size mismatch: have=%ld expect=%zu\n", ...);
        close(fd);
        return -1;
    }
    map = mmap(NULL, expect, PROT_READ | PROT_WRITE, MAP_SHARED, fd, 0);
```

**机理**：镜像必须**恰好** 279552 字节，否则拒绝挂载——避免把普通文本文件当磁盘读入导致结构错乱。

### 3.7 Shell 到 `disk_io` 的调用链（串联动态演示）

用户在 Shell 输入 `format`：

```
main.c: dispatch_command("format")
  → shell_format()
      → vfs_format_disk(path)          [vfs.c]
          → format(path)               [format.c]
              → disk_create()
              → disk_write_block(0, …)  … 写各元数据块
              → disk_save(path)         落盘为 testimg/vfs_disk.img
      → vfs_mount(path)
          → fs_mount() → disk_load(path)   再次 mmap 同一文件
```

`shell_format` 片段：

```408:422:src/main.c
static int shell_format(const char *path)
{
    if (g_mounted) { ui_err("Umount first before format"); return -1; }
    if (vfs_format_disk(use_path) != 0) { ui_err("Format failed"); return -1; }
    ui_ok("Format complete");
    if (vfs_mount(use_path) != 0) { ui_err("Mount after format failed"); return -1; }
    // … 初始化用户、内核子系统等
}
```

讲解时强调：**用户只打一条命令，最底层一定发生 `disk_create` + 多次 `disk_write_block` + `disk_save`。**

## 六、常见问题（答辩/自学）

**Q1：为什么不直接用 `fopen` 读写 `vfs_disk.img`？**  
因为文件系统代码要假设「块设备语义」：固定块大小、块号寻址。`disk_io` 统一这一层后，上层 `buf`、`journal` 才能按 industry 方式设计。

**Q2：`g_disk_mem` 和镜像文件是什么关系？**  
`format` 阶段多为 `calloc` 内存盘，最后 `disk_save` 写出文件；`mount` 阶段 `disk_load` 用 `mmap` 让指针直接指向文件内容，减少拷贝。

**Q3：273KB 够吗？**  
课程设计验证的是**机制**（块组、inode、日志），不是容量；原理与 ext4 相同，只是规模缩小。

**Q4：块 0、块 1 里具体二进制是什么？**  
块 0：`UPFSBOOT` + 尾部指针；块 1：完整 `SuperBlock` 结构（512 字节）。细节在第二部分结合 `SuperBlock` 字段逐项讲。
