# Unix File System Simulator --- UPFS

## 一、项目概述

本项目（UPFS）是一个面向操作系统课程设计的综合性模拟系统，采用纯 C 语言（C17 标准）实现，无任何外部依赖，仅需要标准 C 库与 POSIX 线程库即可编译运行。项目在用户态完整模拟了一个 UNIX 风格的操作系统内核，涵盖文件系统、内存管理、进程管理、虚拟机执行引擎、调度器、进程间通信、多用户管理等核心功能，并提供了一个交互式 Shell 和基于浏览器的图形化桌面环境。

整个系统可划分为五个层次：**虚拟磁盘与文件系统层**、**操作系统内核层**、**用户管理层**、**工具链层**和**网络服务层**。各层次之间通过清晰的接口解耦，上层通过虚拟文件系统（VFS）抽象层访问底层文件系统，内核层通过系统调用接口桥接虚拟机程序与文件系统操作。

---

## 二、运行环境

**操作系统要求**：Linux 或 macOS（任意支持 POSIX 的系统）。

**编译工具**：GCC 或 Clang，需支持 C17 标准。

**构建方式**：使用 CMake 构建系统。

```bash
cmake -B build -S . -DCMAKE_C_FLAGS="-D_GNU_SOURCE"
cmake --build build
```

**运行方式**：

- **交互式 Shell 模式**：直接运行可执行文件，进入命令行交互界面，可执行文件系统操作、运行程序、管理用户等。
- **网络服务模式**：以 `--serve` 参数启动后，系统以守护进程方式运行，同时提供 HTTP 网页服务（端口 8080）和原始 TCP 终端服务（默认端口 4096），支持多个浏览器或终端客户端并发接入。

```bash
./build/bin/OS_design          # 交互式 Shell
./build/bin/OS_design --serve  # 网络服务模式
```

在网络服务模式下，浏览器访问 `http://localhost:8080` 即可打开一个完整的桌面环境，包含终端、文件管理器、文本编辑器和系统监视器四个应用程序。

## 三、磁盘与文件系统结构（用数据说明）

### 3.1 磁盘总体布局

整个虚拟磁盘由 **546 个物理块**构成，每块 **512 字节**，磁盘总容量为 **273 KB**。磁盘在逻辑上划分为四个区域：

| 区域 | 起始块号 | 块数 | 大小 | 用途 |
|------|---------|------|------|------|
| 引导块 | 0 | 1 | 512 B | 存放启动标识和关键块号 |
| 超级块 | 1 | 1 | 512 B | 记录文件系统全局元数据 |
| 块组区 | 2 ~ 513 | 512 | 256 KB | 8 个块组，存放全部文件数据与目录数据 |
| 日志区 | 514 ~ 545 | 32 | 16 KB | 元数据事务日志，保障崩溃一致性 |

### 3.2 块组结构

块组区采用类似 ext2 的设计，共 **8 个块组**，每组 **64 块**。每个块组的第一块是**锚块**（存放空闲块栈和链表头），其余 **63 块**为数据块。因此全部数据块共 **504 块**，合计 **252 KB** 可用于存储文件和目录。

每个块组的空闲块管理采用**成组链接法**：锚块中维护一个容量为 50 的空闲块号栈；当栈满时，将栈中所有块号写入一个新的空闲块并通过链指针串联，形成多级链表。分配时从栈顶弹出，栈空则从链表中加载下一组。这种方式兼顾了分配效率与空间利用率。

### 3.3 索引节点（inode）管理

项目没有预留固定的 inode 区域，而是采用了类似 XFS 的**动态 inode 分块**策略。每个 inode 占 **32 字节**，一个数据块可容纳 **16 个 inode**，称为一个"inode 块"。当需要新的 inode 时，系统从数据区分配一个新块作为 inode 块。理论上，全部 504 个数据块都用来存放 inode 时，最多可容纳 **8064 个 inode**。

为了快速定位任意 inode 号对应的物理位置，系统维护两棵 **B+ 树**：

- **位置树**（Loc Tree）：键为 inode 号，值为该 inode 所在的物理块号和块内槽位偏移。通过这棵树可以在对数时间内找到任意 inode 的磁盘位置。
- **空闲树**（Chunk Tree）：键为 inode 块的块号，值为该块内 16 个槽位的空闲位图。分配新 inode 时，先在空闲树中找到有空余槽位的块，再从中取用。

超级块中还维护了一个 **128 项的空闲 inode 栈**，回收 inode 时优先压入此栈，分配时优先从此栈弹出，以减少 B+ 树操作的频率。

### 3.4 文件数据映射

每个磁盘 inode 内嵌一个 **Extent 结构体**（8 字节），记录逻辑块号、物理块号和连续长度三个字段。对于小文件，一条 Extent 即可直接描述全部数据块的映射关系，无需额外存储。

当文件增长到需要多条映射记录时，系统自动将映射关系提升到 **Extent B+ 树**：叶节点块最多存放 **62 条 Extent 记录**。写入时，如果新分配的物理块恰好与已有区段在逻辑和物理上连续，系统会自动**合并相邻区段**，从而减少元数据占用并提高查找效率。

### 3.5 块缓存

系统在内存中维护一个全局块缓存，共 **1024 个缓存槽**（每槽 512 字节，合计 **512 KB**），采用 **LRU（最近最少使用）+ 哈希表**双索引结构。读取磁盘块时先查哈希表，命中则直接返回缓存数据；未命中则从磁盘读入并插入缓存。写操作支持**延迟写**（标记脏位后由后续刷盘统一回写），减少磁盘访问次数。

### 3.6 日志与崩溃恢复

日志区共 32 块，第一块为日志头，记录当前序列号、日志条目数和提交状态。其余 31 块为日志数据区，每块存放一条日志记录（包含目标块号和完整的 512 字节块数据）。

每次修改元数据（超级块、inode 块、锚块、目录块等），系统先将修改前后的数据写入日志区，然后标记提交完成，最后才回写到原始位置。如果在回写过程中发生中断，下次挂载时日志重放模块会检测到未完成的事务，自动重新执行日志中记录的写操作，从而保证文件系统的一致性。

### 3.7 目录与路径

每个目录项 **16 字节**，其中 2 字节存储 inode 号，14 字节存储文件名（含终止符，因此有效文件名最长 **13 个字符**）。一个块可存放 **32 个目录项**。

路径解析函数支持绝对路径、相对路径以及 `.`（当前目录）和 `..`（父目录）的处理。

### 3.8 打开文件管理

系统实现了经典的**两级打开文件表**结构：

- **用户打开文件表**：每个用户最多同时打开 **20 个文件**，表项记录独立的读写位置。
- **系统打开文件表**：全局共 **40 个表项**，多个用户表项可指向同一系统表项，系统表项记录引用计数和模式标志。
- **活跃 inode 缓存**：最大容纳 **2048 个**内存 inode，采用 128 桶哈希表索引。每个内存 inode 附带一把读写锁，支持多线程并发访问。

### 3.9 虚拟文件系统（VFS）

系统设计了一个可插拔的虚拟文件系统抽象层。每种文件系统实现需要注册三张操作表：inode 操作表（路径解析、创建目录、权限检查等）、文件操作表（打开、读写、关闭等）和超级块操作表（挂载、卸载、格式化、同步）。上层 Shell 和系统调用统一通过 VFS 接口访问文件系统，实现了接口与实现的解耦。

## 四、核心数据结构

本节介绍系统中各层次的关键数据结构，说明每个结构体的字段含义、所占空间和设计意图。

### 4.1 超级块（SuperBlock，512 字节）

超级块是文件系统的"身份证"，记录全局元数据，挂载时首先读入内存。

| 字段 | 类型 | 大小 | 说明 |
|------|------|------|------|
| 魔数 | 32 位无符号整数 | 4 B | 固定值，ASCII 对应 "UPFS"，用于识别文件系统类型 |
| 已用 inode 总数 | 32 位无符号整数 | 4 B | 当前文件系统中已分配的 inode 数量 |
| 空闲 inode 数 | 32 位无符号整数 | 4 B | 空闲 inode 缓存栈中的可用 inode 数 |
| 下一 inode 号 | 32 位无符号整数 | 4 B | 下次分配 inode 时的起始搜索编号 |
| 总块数 | 32 位无符号整数 | 4 B | 数据区的总块数（504） |
| 空闲块数 | 32 位无符号整数 | 4 B | 所有块组的空闲块之和 |
| 块组数 | 16 位无符号整数 | 2 B | 固定为 8 |
| 位置树根块号 | 16 位无符号整数 | 2 B | inode 位置 B+ 树的根节点所在块号 |
| 位置树层级 | 16 位无符号整数 | 2 B | 位置 B+ 树的当前高度 |
| 空闲树根块号 | 16 位无符号整数 | 2 B | inode 空闲 B+ 树的根节点所在块号 |
| 空闲树层级 | 16 位无符号整数 | 2 B | 空闲 B+ 树的当前高度 |
| 空闲 inode 栈顶 | 16 位无符号整数 | 2 B | 栈顶索引，空栈时为特殊标记值 |
| 空闲 inode 栈 | 32 位整数数组 | 128 B | 最近回收的 32 个 inode 号，优先从此栈分配 |
| 块组描述表 | 结构体数组 | 64 B | 8 个块组描述符，各含锚块号、数据起始块号和数据块数 |
| 填充 | 字节数组 | 补齐至 512 B | 保证超级块恰好占满一个物理块 |

### 4.2 磁盘索引节点（DiskINode，32 字节）

每个文件或目录在磁盘上对应一个 32 字节的索引节点，一个物理块可存放 16 个。

| 字段 | 类型 | 大小 | 说明 |
|------|------|------|------|
| 类型与权限 | 16 位无符号整数 | 2 B | 高位标识目录或普通文件，低 9 位为所有者/组/其他的读写执行权限 |
| 硬链接计数 | 16 位无符号整数 | 2 B | 指向该 inode 的目录项数量，降为 0 时释放数据块和 inode |
| 文件长度 | 32 位无符号整数 | 4 B | 文件包含的有效字节数 |
| 所有者用户号 | 16 位无符号整数 | 2 B | 创建该文件的用户的 UID |
| 所有者组号 | 16 位无符号整数 | 2 B | 创建该文件的用户所属的 GID |
| 内联 Extent | Extent 结构体 | 8 B | 直接记录一段连续的逻辑块→物理块映射（小文件无需额外存储） |
| Extent 树根块号 | 16 位无符号整数 | 2 B | 大文件的 Extent B+ 树根节点块号，小文件为 0 |
| Extent 树层级 | 16 位无符号整数 | 2 B | B+ 树高度，叶节点为 0 层 |
| 保留字段 | — | 8 B | 对齐填充至 32 字节边界 |

### 4.3 Extent 区段（8 字节）

Extent 是文件数据映射的基本单元，描述一段连续的逻辑块到物理块的对应关系。

| 字段 | 类型 | 大小 | 说明 |
|------|------|------|------|
| 起始逻辑块号 | 32 位无符号整数 | 4 B | 该区段在文件内部的起始逻辑块偏移 |
| 起始物理块号 | 16 位无符号整数 | 2 B | 该区段在磁盘上的起始物理块号 |
| 连续块数 | 16 位无符号整数 | 2 B | 该区段覆盖的连续物理块数量 |

一条 Extent 记录可描述最多 65535 个连续块的映射。当文件顺序写入时，相邻的 Extent 会被自动合并，通常只需极少的记录即可覆盖整个文件。

### 4.4 目录项（DirEntry，16 字节）

| 字段 | 类型 | 大小 | 说明 |
|------|------|------|------|
| inode 号 | 16 位无符号整数 | 2 B | 该目录项所指向的文件或子目录的 inode 号，为 0 表示空槽 |
| 文件名 | 字符数组 | 14 B | 文件名字符串，含终止符，有效名最长 13 字符 |

一个 512 字节的数据块可存放 32 个目录项。删除文件时只需将目录项的 inode 号置零，无需搬移后续条目。

### 4.5 块组锚块（BgAnchorDisk，512 字节）

每个块组的第一个块为锚块，是成组链接法空闲管理的核心。

| 字段 | 类型 | 大小 | 说明 |
|------|------|------|------|
| 魔数 | 32 位无符号整数 | 4 B | 锚块标识，用于校验数据完整性 |
| 空闲块计数 | 32 位无符号整数 | 4 B | 该块组当前剩余的空闲块总数 |
| 栈内计数 | 16 位无符号整数 | 2 B | 空闲块栈中当前存放的块号数量 |
| 链表头指针 | 16 位无符号整数 | 2 B | 链表中下一组空闲块所在的寄存块号，为 0 表示无后继 |
| 空闲块栈 | 16 位整数数组 | 100 B | 最多存放 50 个空闲块号，分配时从栈顶弹出 |
| 填充 | 字节数组 | 补齐至 512 B | 保证锚块恰好占满一个物理块 |

### 4.6 内存索引节点（MemINode，40+ 字节）

文件系统在内存中维护的 inode 缓存条目，在磁盘 inode 基础上附加了运行时管理信息。

| 字段 | 类型 | 大小 | 说明 |
|------|------|------|------|
| 磁盘 inode 副本 | DiskINode 结构体 | 32 B | 从磁盘读入的完整 inode 数据 |
| inode 编号 | 16 位无符号整数 | 2 B | 该 inode 在文件系统中的编号 |
| 引用计数 | 16 位无符号整数 | 2 B | 当前有多少处代码持有该 inode 的引用 |
| 状态标志 | 8 位无符号整数 | 1 B | 脏标志（已修改待回写）和锁定标志 |
| 读写锁 | 线程读写锁 | 平台相关 | 附加在哈希节点上，支持并发读和互斥写 |

每个缓存条目被包装在哈希节点中，通过哈希链串接到 128 桶的哈希表中，实现按 inode 号的快速查找。

### 4.7 用户打开文件表项（OpenFileTable，16 字节）

| 字段 | 类型 | 大小 | 说明 |
|------|------|------|------|
| 读位置 | 32 位无符号整数 | 4 B | 该用户下次读取的起始字节偏移 |
| 写位置 | 32 位无符号整数 | 4 B | 该用户下次写入的起始字节偏移 |
| 打开模式 | 16 位无符号整数 | 2 B | 只读、只写或读写 |
| 锁定标志 | 16 位无符号整数 | 2 B | 是否持有 inode 的读锁或写锁 |
| 指向内存 inode 的指针 | 指针 | 4/8 B | 关联的内存 inode 缓存条目 |

每个用户独立维护最多 20 个这样的表项，不同用户打开同一文件时各自持有独立的读写位置。

### 4.8 进程控制块（PCB）

PCB 是内核中最大的数据结构，包含进程运行所需的全部信息。

| 字段组 | 内容 | 说明 |
|--------|------|------|
| 标识信息 | 进程号、父进程号、进程名（31 字符）、当前状态 | 五种状态：空闲、就绪、运行、阻塞、僵尸 |
| CPU 上下文 | 16 个通用寄存器、程序计数器、标志寄存器、剩余时间片、系统调用参数 | 上下文切换时整体保存和恢复 |
| 内存布局 | 4096 项页表、代码段起止、数据段起止、BSS 结束地址、堆顶指针、栈顶地址 | 每进程最大 16 MB 虚拟地址空间 |
| 文件描述符表 | 16 项，每项记录类型（终端/文件/管道读端/管道写端/命名管道）、关联的文件系统句柄、管道编号 | 支持 fork 时完整复制 |
| 进程关系 | 子进程号数组（最多 64 个）、子进程计数、退出码 | 用于 wait 回收和孤儿过继 |
| IPC 状态 | 信号挂起位掩码、用户信号计数器、共享内存挂载表（最多 4 段） | 每段记录共享内存编号和挂载的虚拟地址 |

### 4.9 管道（Pipe）

| 字段 | 类型 | 大小 | 说明 |
|------|------|------|------|
| 数据缓冲区 | 字节数组 | 512 B | 环形缓冲区，存放管道中的待读取数据 |
| 有效字节数 | 整型 | 4 B | 缓冲区中当前可读的字节数 |
| 读指针 | 整型 | 4 B | 下次读取的起始位置 |
| 写指针 | 整型 | 4 B | 下次写入的起始位置 |
| 读端引用计数 | 整型 | 4 B | 当前持有读端的进程数 |
| 写端引用计数 | 整型 | 4 B | 当前持有写端的进程数 |

读写指针在到达缓冲区末尾时自动回绕到起始位置，形成环形队列。当两端引用计数均降为零时管道自动回收。

### 4.10 虚拟机指令编码格式（32 位定长）

| 位段 | 位宽 | 位置 | 说明 |
|------|------|------|------|
| 操作码 | 8 位 | [31:24] | 标识指令类型，共 22 种 |
| 目标寄存器 | 4 位 | [23:20] | 运算结果写入的寄存器编号（0~15） |
| 源寄存器 1 | 4 位 | [19:16] | 第一操作数寄存器编号 |
| 源寄存器 2 | 4 位 | [15:12] | 第二操作数寄存器编号 |
| 立即数 | 12 位 | [11:0] | 有符号立即数，范围 -2048 ~ +2047 |

所有指令采用统一的 32 位编码格式，简化了取指和译码逻辑。不同指令类型使用不同的字段组合：算术指令使用三个寄存器字段，加载立即数使用目标寄存器和立即数字段，跳转指令使用立即数作为相对偏移量。

### 4.11 UPX 可执行文件头（24 字节）

| 字段 | 类型 | 大小 | 说明 |
|------|------|------|------|
| 魔数 | 字符数组 | 4 B | 固定为 "UPX\0"，用于识别文件格式 |
| 入口指令索引 | 32 位无符号整数 | 4 B | 程序从第几条指令开始执行 |
| 代码段大小 | 32 位无符号整数 | 4 B | 代码段字节数 |
| 数据段大小 | 32 位无符号整数 | 4 B | 已初始化数据段字节数 |
| BSS 段大小 | 32 位无符号整数 | 4 B | 未初始化数据段字节数（加载时清零） |
| 栈大小 | 32 位无符号整数 | 4 B | 进程栈的分配大小（默认 4096 字节） |

文件头之后依次存放代码段和数据段的二进制内容。加载器根据各段大小计算需要的物理页数，分配内存并将内容复制到对应的虚拟地址区间。

### 4.12 日志头（JournalHeader，16 字节）

| 字段 | 类型 | 大小 | 说明 |
|------|------|------|------|
| 魔数 | 32 位无符号整数 | 4 B | 日志区标识 |
| 序列号 | 32 位无符号整数 | 4 B | 每次提交事务时递增，用于区分不同事务 |
| 日志条目数 | 32 位无符号整数 | 4 B | 当前事务包含的日志记录数量 |
| 提交标志 | 32 位无符号整数 | 4 B | 为 1 表示事务已完成回写，为 0 表示需要重放 |

### 4.13 内核全局共享状态（KernelShared）

在网络服务模式下，多个会话子进程通过共享内存段访问同一个内核状态。

| 字段组 | 大小 | 说明 |
|--------|------|------|
| 物理内存 | 128 MB | 全部进程共享的模拟物理内存 |
| 页位图 | 4 KB | 32768 位，标记每个物理页的分配状态 |
| 进程表 | 64 个 PCB | 系统中全部进程的控制块数组 |
| 就绪队列 | 65 项环形数组 | 时间片轮转调度的就绪进程索引 |
| 管道表 | 16 个管道 | 所有匿名管道和命名管道的数据与状态 |
| IPC 表 | 信号量 + 消息队列 + 共享内存 + 命名管道 | 全部进程间通信资源 |
| 互斥锁 | 线程互斥锁 | 多进程并发访问时的同步保护 |

## 五、内核层结构

### 5.1 物理内存管理

模拟器提供 **128 MB** 的物理内存空间（一个连续的字节数组），划分为 **32768 页**（每页 **4 KB**）。其中前 **4096 页**（16 MB）保留给内核使用，其余页面供用户进程分配。

页面分配器使用**位图法**：每一位对应一个物理页，1 表示已分配，0 表示空闲。分配连续 N 页时，算法在位图中线性扫描寻找连续 N 个空闲位。

### 5.2 虚拟机（CPU）

项目实现了一个 **32 位 RISC 指令集虚拟机**，支持 **22 条指令**，指令编码为固定 32 位格式。虚拟机具有 16 个通用寄存器（其中第 15 号寄存器作为栈指针使用）、一个程序计数器和一个标志寄存器。

虚拟机支持**虚拟地址到物理地址的页表转换**：每个进程拥有独立的页表（最多 4096 个虚拟页，对应 **16 MB 虚拟地址空间**），虚拟机在执行访存指令时自动通过页表完成地址翻译。

执行引擎采用经典的**取指—译码—执行**循环。每执行一条指令，时间片计数器减一；减到零则产生时间片中断，触发调度切换。

### 5.3 进程管理

系统最多支持 **64 个并发进程**，每个进程由一个进程控制块（PCB）描述，包含进程标识、状态、CPU 上下文快照、页表、文件描述符表（每进程最多 **16 个**）、子进程列表、环境变量指针等信息。

进程生命周期涵盖五种状态：空闲、就绪、运行、阻塞和僵尸。系统提供了完整的进程操作原语：

- **创建**：分配 PCB 和初始物理页。
- **Fork**：复制父进程的全部页表和内存内容，子进程获得独立的地址空间。
- **Exec**：加载可执行文件（UPX 格式），建立代码段、数据段、BSS 段和栈段的内存映射。
- **Wait**：父进程等待子进程退出，回收僵尸进程。
- **Exit**：释放进程资源，将子进程过继给 init 进程以防止孤儿僵尸。

### 5.4 调度器

采用**时间片轮转**调度算法，时间片大小为 **100 条指令**。就绪队列使用环形数组实现。每次时间片耗尽后，当前进程重新加入就绪队列尾部，调度器从队列头部取出下一个就绪进程投入运行。

调度器在每次调度间隙自动回收父进程为 init 的僵尸进程。系统还提供了协作式让出机制，在管道读写和信号量等待等阻塞操作中，进程主动让出 CPU 给其他就绪进程，避免忙等。

### 5.5 系统调用

虚拟机通过专门的 SYSCALL 指令触发系统调用。调用号通过指令立即数传递，参数和返回值使用寄存器传递。系统共实现了 **30 余个系统调用**，涵盖以下类别：

- **进程类**：退出、派生、执行、等待、获取进程号
- **文件类**：创建、打开、读、写、关闭、定位、删除、查询属性
- **目录类**：获取工作目录、切换目录、创建目录
- **内存类**：堆扩展
- **环境变量类**：获取、设置、删除环境变量
- **IPC 类**：管道、信号发送与接收、信号量操作、消息队列收发、共享内存映射与解除映射、命名管道创建与打开
- **工具类**：内置编辑器调用、汇编器调用

### 5.6 进程间通信

系统实现了五种完整的进程间通信机制：

**匿名管道**：最多 **16 条**并发管道，每条管道内部维护一个 **512 字节**的环形缓冲区，支持引用计数的读端/写端管理。读取空管道时进程自动让出 CPU 等待写端写入数据，实现了生产者—消费者式的同步。Shell 层面支持 **2~8 级**管道串联。

**命名管道（FIFO）**：最多 **8 条**命名管道，通过路径标识。底层复用匿名管道的缓冲机制，但具备文件系统路径，不相关的进程也可以通过路径打开同一个管道进行通信。

**信号量**：最多 **16 个**信号量，采用键值方式获取。支持 P 操作（等待/减一）和 V 操作（释放/加一）。P 操作在信号量值为零时自动让出 CPU 等待。

**消息队列**：最多 **8 个**消息队列，每队列最多缓存 **32 条消息**，每条消息最大 **128 字节**。支持按消息类型接收，实现了有选择的进程间消息传递。

**共享内存**：最多 **8 个**共享内存段，每段最多 **4 页**（16 KB）。进程通过映射操作将共享内存段挂载到自己的虚拟地址空间中，多个进程可同时访问同一块物理内存，实现高效的数据共享。

**信号机制**：支持 SIGKILL（强制终止）、SIGTERM（请求终止）和 SIGUSR1（用户自定义）三种信号。信号以位掩码方式挂起，在调度切换时投递。收到 KILL 或 TERM 信号的进程立即转为僵尸状态。

## 六、用户管理与环境变量

### 6.1 多用户支持

系统最多支持 **10 个**用户账户，用户数据库存储在虚拟磁盘的 `/etc/passwd` 文件中，格式为"用户名:UID:密码哈希:盐值:主目录"的冒号分隔文本。

密码安全方面，系统采用**加盐迭代哈希**：随机生成 16 字节盐值，将盐值与密码拼接后经过 **10000 轮**混合运算，产生 32 字节（256 位）的哈希值。盐值从 `/dev/urandom` 获取，在不可用时退化为基于时间戳和计数器的伪随机序列。

首次格式化磁盘时，系统会自动创建 `/bin`、`/home`、`/root`、`/etc` 等标准 POSIX 目录结构，并引导用户创建第一个账户。用户登录后自动切换到其主目录，加载该用户的环境变量。

### 6.2 UNIX 权限模型

每个文件和目录的 inode 中存储了所有者 UID、组 GID 和 9 位权限掩码（所有者/组/其他各三位读写执行权限）。权限检查遵循 UNIX 规则：先匹配所有者，再匹配组，最后匹配其他用户。UID 为 0 的 root 用户绕过所有权限检查。

### 6.3 环境变量

环境变量分为系统级和用户级两层。系统级变量存储在 `/etc/environment` 文件中，由 root 用户管理；用户级变量存储在各用户主目录下的 `.env` 文件中。查询变量时优先返回用户级值，用户级未定义则回退到系统级。每层最多存储 **64 个**变量，键名最长 63 字符，值最长 255 字符。

## 七、工具链

### 7.1 两遍汇编器

系统内置了一个完整的两遍汇编器，可以将汇编源文件编译为 UPX 可执行文件。汇编器支持全部 22 条指令的助记符，以及 `.text`、`.data`、`.bss`、`.entry`、`.word`、`.space`、`.ascii`、`.asciz`、`.stack` 等伪指令。

第一遍扫描收集所有标签地址和段大小；第二遍生成机器码，自动计算跳转偏移和地址引用。最终输出 UPX 格式的二进制文件。

### 7.2 C 语言编译器

系统还内置了一个 C 语言子集的编译器，支持整型变量、数组、函数定义与调用、条件分支、循环、指针运算等基本语法。编译流程为：词法分析 → 语法分析（生成抽象语法树）→ 代码生成（输出汇编源码）→ 汇编器（生成 UPX 二进制）。用户在 Shell 中使用 `cc` 命令即可一键完成从 C 源码到可执行文件的编译。

### 7.3 内置文本编辑器

系统内置了一个类 vim 的全屏文本编辑器，支持终端 raw 模式下的光标移动、文本编辑、行号显示、语法高亮等功能。用户可以在 Shell 中使用 `vim` 命令编辑虚拟磁盘中的文件。

### 7.4 可执行文件格式（UPX）

UPX 是本项目定义的可执行文件格式，文件头包含 4 字节魔数 "UPX\0"、入口指令索引、代码段大小、数据段大小、BSS 段大小和栈大小共 24 字节，随后依次存放代码段和数据段的二进制内容。格式化磁盘时，系统自动向 `/bin/` 目录注入预编译的演示程序。

## 八、网络服务与桌面环境

在 `--serve` 模式下，系统启动一个多进程网络服务器：

- **HTTP 服务器**（端口 8080）：响应浏览器请求，返回内嵌的单页面 Web 应用。
- **WebSocket 终端**（路径 `/ws/N`）：为每个浏览器标签页提供独立的终端会话，通过 WebSocket 协议实时传输输入输出。
- **JSON API**（路径 `/api`）：为文件管理器、文本编辑器和系统监视器提供文件列表、文件读写、进程状态等数据接口。
- **原始 TCP 终端**（默认端口 4096）：兼容传统终端客户端的接入。

所有会话子进程通过 mmap 共享同一个磁盘镜像文件，每次命令执行前重新加载超级块元数据以获取最新状态，命令完成后同步回写，从而实现多终端之间的数据一致性。

Web 桌面环境提供了终端、文件管理器（Finder）、文本编辑器（TextEdit）和系统监视器（Activity Monitor）四个窗口化应用程序，具有停靠栏、启动屏幕和窗口管理等图形界面特性。

## 十、重点函数源代码与说明

### 10.1 磁盘格式化（format）

磁盘格式化是文件系统从无到有的第一步。该函数依次完成内存分配、引导块写入、块组初始化、日志区清零、动态 inode 映射初始化和超级块写入，最后将根目录建立在固定位置并把内存盘持久化到磁盘镜像文件。

```c
int format(const char *disk_path)
{
    SuperBlock sb;

    if (disk_path == NULL || disk_path[0] == '\0') {
        FORMAT_LOG("错误：磁盘路径为空");
        return -1;
    }

    if (disk_create() != 0) {
        FORMAT_LOG("错误：内存分配失败");
        return -1;
    }

    if (init_boot_block() != 0) {
        FORMAT_LOG("错误：引导块写入失败");
        disk_shutdown();
        return -1;
    }

    if (bg_format_init() != 0) {
        FORMAT_LOG("错误：块组初始化失败");
        disk_shutdown();
        return -1;
    }

    if (journal_init_format() != 0) {
        FORMAT_LOG("错误：日志区初始化失败");
        disk_shutdown();
        return -1;
    }

    if (inomap_format_init(ROOT_DIR_BLOCK) != 0) {
        FORMAT_LOG("错误：动态 inode 映射初始化失败");
        disk_shutdown();
        return -1;
    }

    memset(&sb, 0, sizeof(sb));
    sb.s_magic = VFS_MAGIC;
    if (bg_fill_super(&sb) != 0) {
        FORMAT_LOG("错误：超级块汇总失败");
        disk_shutdown();
        return -1;
    }

    if (bg_sync() != 0) {
        FORMAT_LOG("错误：块组 anchor 同步失败");
        disk_shutdown();
        return -1;
    }

    if (disk_write_block(SUPERBLOCK_BLOCKNO, &sb) != 0) {
        FORMAT_LOG("错误：超级块写入失败");
        disk_shutdown();
        return -1;
    }

    if (init_root_directory() != 0) {
        FORMAT_LOG("错误：根目录初始化失败");
        disk_shutdown();
        return -1;
    }

    if (disk_save(disk_path) != 0) {
        FORMAT_LOG("错误：镜像文件保存失败");
        disk_shutdown();
        return -1;
    }

    disk_shutdown();
    return 0;
}
```

这个函数体现了文件系统初始化时各子系统之间的依赖顺序：必须先建立块组的空闲块管理，才能为 inode B+ 树分配元数据块；B+ 树建立之后要通过块组同步将最新的空闲块状态写回锚块，否则后续挂载时锚块中的空闲块栈会包含已被 B+ 树占用的块号。这个精确的初始化时序是保证磁盘数据一致性的关键。

### 10.2 块分配（bg_balloc_for）

块分配是文件系统中最频繁的操作之一。该函数根据传入的 inode 号提示确定优先分配的块组——尽量在 inode 所属的块组内分配数据块以提高局部性。如果该块组已满，则循环尝试下一个块组。

```c
int bg_balloc_for(uint16_t ino_hint)
{
    int start = bg_hint_to_group(ino_hint);
    int bg;
    BgRuntime *rt;
    uint16_t blk;

    for (int pass = 0; pass < BG_COUNT; pass++) {
        bg = (start + pass) % BG_COUNT;
        rt = &g_bg[bg];
        if (rt->block_free == 0)
            continue;
        if (rt->free_stack_count == 0) {
            if (bg_reload_free_stack(rt) != 0)
                continue;
        }
        rt->free_stack_count--;
        blk = rt->free_stack[rt->free_stack_count];
        rt->block_free--;
        return (int)blk;
    }
    return -1;
}
```

当空闲块栈为空但该块组仍有空闲块时，分配函数会调用栈重装过程：从链表中读取下一组空闲块号填充到栈中，然后再执行弹出操作。这种两级结构使得常态分配只需一次栈操作，只有栈耗尽时才需要一次磁盘读取，兼顾了效率和空间利用率。

### 10.3 路径解析（namei）

路径解析是文件系统中所有操作的入口。该函数将一个路径字符串逐个分量地解析为最终的内存 inode 指针。绝对路径从根目录开始，相对路径从当前用户的工作目录开始。

```c
MemINode *namei(const char *path)
{
    MemINode   *ip;
    const char *p;
    char        comp[MAX_FILENAME_LEN + 1];
    uint16_t    start_ino;

    if (path == NULL || g_cur_user == NULL || fs_get_superblock() == NULL)
        return NULL;

    start_ino = (path[0] == '/') ? ROOT_INODE_NO : g_cur_user->u_cdir;
    ip = iget(start_ino);
    if (ip == NULL)
        return NULL;

    if (path[0] == '/') {
        p = path_skip_slash(path + 1);
        if (*p == '\0')
            return ip;
        p = path + 1;
    } else {
        p = path;
    }

    while (*p != '\0') {
        uint16_t next_ino;

        p = path_next_component(p, comp);
        if (p == NULL) { iput(ip); return NULL; }
        if (comp[0] == '\0') break;

        if (!inode_is_dir(ip)) { iput(ip); return NULL; }

        if (strcmp(comp, ".") == 0) {
            if (path_is_last_component(p)) return ip;
            p = path_skip_slash(p);
            continue;
        }

        if (strcmp(comp, "..") == 0) {
            MemINode *parent_ip;
            if (dir_lookup(ip, "..", &next_ino) != 0) { iput(ip); return NULL; }
            parent_ip = iget(next_ino);
            iput(ip);
            ip = parent_ip;
            if (ip == NULL) return NULL;
            if (path_is_last_component(p)) return ip;
            p = path_skip_slash(p);
            continue;
        }

        if (dir_lookup(ip, comp, &next_ino) != 0) { iput(ip); return NULL; }

        if (path_is_last_component(p)) {
            MemINode *target = iget(next_ino);
            iput(ip);
            return target;
        }

        {
            MemINode *next_ip = iget(next_ino);
            iput(ip);
            ip = next_ip;
            if (ip == NULL) return NULL;
        }
        p = path_skip_slash(p);
    }

    return ip;
}
```

该函数的核心逻辑是一个循环：每次取出路径中的下一个分量名，在当前目录的 inode 中查找该名称对应的子目录项，获取子目录的 inode 号并加载到内存。遇到 `.` 时跳过，遇到 `..` 时回退到父目录。每一步都通过引用计数管理内存 inode 的生命周期——进入新目录时获取引用，离开旧目录时释放引用。

### 10.4 虚拟机单步执行（cpu_step）

这是整个系统最核心的执行引擎。每次调用执行一条虚拟机指令，实现了完整的取指、译码和执行过程。

```c
int cpu_step(CPUContext *ctx)
{
    if (ctx == NULL) return 1;
    ctx->ticks_left--;

    uint32_t inst = cpu_read32(ctx, ctx->pc * 4);
    uint32_t op = CPU_OPCODE(inst);
    uint32_t rd = CPU_RD(inst);
    uint32_t rs1 = CPU_RS1(inst);
    uint32_t rs2 = CPU_RS2(inst);
    int32_t  imm = CPU_IMM12(inst);

    uint32_t *r = ctx->regs;

    switch (op) {
    case OP_HALT:
        ctx->sycall_halt = 1;
        ctx->syscall_no = SYSCALL_EXIT;
        return 1;

    case OP_MOVI:
        if (rd < CPU_NUM_REGS) r[rd] = (uint32_t)(int32_t)imm;
        ctx->pc++;
        break;

    case OP_ADD:
        if (rd < CPU_NUM_REGS && rs1 < CPU_NUM_REGS && rs2 < CPU_NUM_REGS)
            r[rd] = r[rs1] + r[rs2];
        ctx->pc++;
        break;

    case OP_LD:
        if (rd < CPU_NUM_REGS && rs1 < CPU_NUM_REGS) {
            uint32_t addr = r[rs1] + (uint32_t)imm;
            r[rd] = cpu_read32(ctx, addr);
        }
        ctx->pc++;
        break;

    case OP_CMP:
        if (rs1 < CPU_NUM_REGS && rs2 < CPU_NUM_REGS)
            ctx->flags = (r[rs1] == r[rs2]) ? CPU_FLAG_ZF : 0;
        ctx->pc++;
        break;

    case OP_JZ:
        if (ctx->flags & CPU_FLAG_ZF)
            ctx->pc = (uint32_t)((int32_t)ctx->pc + imm);
        else
            ctx->pc++;
        break;

    case OP_CALL:
        r[CPU_REG_SP] -= 4;
        cpu_write32(ctx, r[CPU_REG_SP], ctx->pc + 1);
        ctx->pc = (uint32_t)((int32_t)ctx->pc + imm);
        break;

    case OP_SYSCALL:
        ctx->sycall_halt = 2;
        ctx->syscall_no = (uint32_t)imm;
        ctx->pc++;
        return 1;

    /* ... 其余指令省略，结构相同 ... */
    default:
        ctx->pc++;
        break;
    }

    if (ctx->ticks_left == 0)
        return 1;
    return 0;
}
```

函数首先通过程序计数器从虚拟内存中取出当前指令（经过页表地址翻译），然后从 32 位指令中提取操作码、目标寄存器、源寄存器和立即数字段。接下来通过一个大的分支结构执行对应的操作——算术运算直接修改寄存器值，访存指令通过页表翻译后访问物理内存，跳转指令修改程序计数器，系统调用指令则设置中断标志并返回，由上层调度器将控制权转交给系统调用处理函数。

当时间片计数器减到零时，函数返回 1 通知调度器执行上下文切换。

### 10.5 进程 Fork

进程派生是操作系统中最精妙的操作之一。该函数完整复制父进程的地址空间、CPU 状态和文件描述符表，创建出一个几乎相同的子进程。

```c
int proc_fork(void)
{
    PCB *parent = proc_current();
    if (parent == NULL) return -1;

    PCB *child = proc_alloc();
    if (child == NULL) return -1;

    for (int i = 0; i < MEM_MAX_PROCESS_PAGES; i++) {
        if (parent->p_page_table[i] != 0) {
            int new_page = mem_alloc_pages(1);
            if (new_page < 0) { proc_free(child); return -1; }
            child->p_page_table[i] = (uint32_t)new_page;
            uint32_t src_phys = parent->p_page_table[i] * MEM_PAGE_SIZE;
            uint32_t dst_phys = (uint32_t)new_page * MEM_PAGE_SIZE;
            mem_copy(dst_phys, src_phys, MEM_PAGE_SIZE);
        }
    }

    memcpy(&child->p_cpu, &parent->p_cpu, sizeof(CPUContext));
    child->p_cpu.ticks_left = CPU_TIMESLICE;

    child->p_ppid = parent->p_pid;
    child->p_text_start = parent->p_text_start;
    child->p_text_pages = parent->p_text_pages;
    child->p_data_start = parent->p_data_start;
    child->p_data_pages = parent->p_data_pages;
    child->p_bss_end = parent->p_bss_end;
    child->p_heap_brk = parent->p_heap_brk;
    child->p_stack_top = parent->p_stack_top;
    child->p_cwd_ino = parent->p_cwd_ino;
    strncpy(child->p_name, parent->p_name, PROC_NAME_LEN - 1);

    for (int i = 0; i < PROC_MAX_FD; i++)
        proc_dup_fd(&child->p_ofile[i], &parent->p_ofile[i]);

    child->p_cpu.regs[0] = 0;

    if (parent->p_child_count < PROC_MAX_COUNT)
        parent->p_children[parent->p_child_count++] = child->p_pid;

    parent->p_cpu.regs[0] = child->p_pid;

    sched_enqueue(child);
    return (int)child->p_pid;
}
```

这个函数的关键在于：遍历父进程的全部页表项，为每个有效的虚拟页分配一个新的物理页并逐字节复制内容，使子进程获得完全独立的内存副本。CPU 上下文完整复制意味着子进程从 fork 返回后将从与父进程相同的位置继续执行。两者唯一的区别在于返回值寄存器——子进程中为 0，父进程中为子进程的进程号。文件描述符也被复制，管道类描述符还会增加对应管道的读/写端引用计数，保证管道在任一端关闭后仍能正常工作。

### 10.6 Extent 块映射（extent_bmap）

文件读写的核心是将逻辑块号转换为物理块号。该函数先尝试在现有映射中查找，若未找到且调用方要求创建，则分配新物理块并插入映射。

```c
int extent_bmap(MemINode *ip, uint32_t lblk, int create, uint16_t *phys_out)
{
    Extent new_e;
    int blk;

    if (ip == NULL || phys_out == NULL) return -1;

    if (extent_lookup(&ip->m_dinode, lblk, phys_out) == 0)
        return 0;

    if (!create) return -1;

    blk = balloc_for(ip->m_inode_no);
    if (blk < 0) return -1;

    if (extent_try_extend_run(ip, lblk, (uint16_t)blk) == 0) {
        *phys_out = (uint16_t)blk;
        return 0;
    }

    new_e.e_lblk = lblk;
    new_e.e_pblk = (uint16_t)blk;
    new_e.e_len = 1;

    if (insert_extent(ip, &new_e) != 0) {
        bfree(blk);
        return -1;
    }

    *phys_out = (uint16_t)blk;
    return 0;
}
```

这个函数体现了 Extent 映射的三层策略：第一步在现有映射中查找，对于已写过的逻辑块直接返回物理块号；第二步尝试扩展当前 Extent 的末尾——如果新分配的物理块恰好和已有区段在逻辑和物理上都相邻，只需将长度加一即可，避免了新增映射记录；第三步才创建新的 Extent 记录并插入到映射结构中。这种优先合并的策略使得顺序写入的文件通常只需很少的映射记录即可覆盖全部数据，显著减少了元数据开销。

### 10.7 日志重放（journal_replay）

挂载文件系统时调用此函数检查日志区，如果发现上次有已提交但未完成回写的事务，则重新执行这些写操作以恢复一致性。

```c
int journal_replay(void)
{
    JournalHeader hdr;
    JournalLogRec rec;
    int i;

    if (journal_read_header(&hdr) != 0)
        return 0;

    if (hdr.jh_committed == 0 || hdr.jh_nlogs == 0)
        return 0;

    for (i = 0; i < (int)hdr.jh_nlogs && i < JOURNAL_MAX_LOG; i++) {
        if (disk_read_block(journal_data_block(i), &rec) != 0)
            return -1;
        if (rec.lr_type != JTXN_LOG)
            continue;
        if (journal_apply_log((int)rec.lr_block, rec.lr_data) != 0)
            return -1;
    }

    hdr.jh_committed = 1;
    hdr.jh_nlogs = 0;
    return disk_write_block(journal_header_block(), &hdr);
}
```

日志重放的逻辑非常简洁：读取日志头，判断是否有已提交的日志记录；如果有，逐条读出并将其中保存的完整块数据写回到原始目标块号的位置。全部重放完成后清空日志头中的记录计数，标记为已完成。由于日志中保存的是完整的块内容而非增量差异，重放过程是幂等的——多次重放不会产生副作用。

### 10.8 文件写入（upfs_write）

文件写入是用户最常使用的操作之一。该函数从用户提供的缓冲区中取出数据，按块边界拆分后逐块写入磁盘。

```c
int upfs_write(int fd, const void *buf, int count)
{
    User          *u;
    OpenFileTable *oft;
    MemINode      *ip;
    const char    *src;
    char           block_buf[BLOCK_SIZE];
    int            total;

    u = dir_get_user();
    if (buf == NULL || count < 0) return -1;
    if (count == 0) return 0;

    oft = oft_get(u, fd);
    if (oft == NULL) return -1;
    if (!(oft->oft_mode & (O_WRONLY | O_RDWR))) return -1;

    ip  = oft->oft_inode;
    src = (const char *)buf;
    total = 0;

    while (total < count) {
        uint32_t lblk;
        uint32_t off;
        uint16_t phys;
        int      chunk;

        lblk = oft->oft_write_pos / (uint32_t)BLOCK_SIZE;
        off  = oft->oft_write_pos % (uint32_t)BLOCK_SIZE;

        if (extent_bmap(ip, lblk, 1, &phys) != 0)
            return (total > 0) ? total : -1;

        if (read_block((int)phys, block_buf) != 0)
            return (total > 0) ? total : -1;

        chunk = (int)(BLOCK_SIZE - off);
        if (chunk > count - total) chunk = count - total;

        memcpy(block_buf + off, src + total, (size_t)chunk);
        if (write_block((int)phys, block_buf) != 0)
            return (total > 0) ? total : -1;

        total += chunk;
        oft->oft_write_pos += (uint32_t)chunk;

        if (oft->oft_write_pos > ip->m_dinode.d_size) {
            ip->m_dinode.d_size = oft->oft_write_pos;
            ip->m_flags |= MINODE_DIRTY;
        }
    }

    return total;
}
```

写入过程的核心是一个循环：根据当前写位置计算逻辑块号和块内偏移，通过 Extent 映射获取物理块号（不存在则自动分配新块），读出物理块的当前内容，将数据拷贝到块内对应位置，再写回磁盘。每次写入只修改一个块内的一段连续区域，跨块时自动进入下一轮循环。如果写位置超过了文件原有大小，则同步更新 inode 中的文件长度字段。

## 十一、创新算法说明

本节详细介绍项目中具有设计创新性的六个核心算法。每个算法均给出设计思路、伪代码和复杂度分析。

### 11.1 成组链接法空闲块管理

**设计思路**：传统的空闲块链表逐块串联，每次分配和回收都需要一次磁盘读写。成组链接法将多个空闲块号"打包"存放，大幅减少管理开销。每个块组的锚块中维护一个容量为 50 的空闲块号栈。分配时直接从栈顶弹出；栈空时，从链表中的下一个寄存块读取一整组块号补充到栈中。回收时将块号压入栈中；栈满时，将整个栈的内容写入被回收的块（使其成为新的寄存块），然后清空栈。

**创新点**：在标准成组链接法基础上，引入了按 inode 所在块组就近分配的策略。分配块时先尝试父 inode 所在的块组，若该组空间不足则顺序轮询其他块组，使文件数据尽可能与其 inode 在同一块组内，提升访问局部性。

**伪代码**：

| 步骤 | 块分配算法（按 inode 就近） |
|:----:|:---|
| 1 | 输入：父 inode 编号 |
| 2 | 通过位置树查找父 inode 所在的物理块号 |
| 3 | 根据物理块号计算其所属块组编号，记为起始组 |
| 4 | **循环** 遍历 8 个块组（从起始组开始，取模轮转）： |
| 5 | 　　若当前块组空闲块数为 0，跳过 |
| 6 | 　　若栈为空，执行**栈补充**操作 |
| 7 | 　　栈顶指针减一，取出栈顶块号作为分配结果 |
| 8 | 　　空闲块数减一，返回该块号 |
| 9 | 所有块组均无空闲块，分配失败 |

| 步骤 | 栈补充操作 |
|:----:|:---|
| 1 | 读取链表头指针指向的寄存块 |
| 2 | 从寄存块中取出头部的条目数 M |
| 3 | 将寄存块中的 M 个块号复制到栈中 |
| 4 | 若 M 小于栈容量，将寄存块本身也加入栈中（M+1 个） |
| 5 | 更新链表头指针为寄存块中记录的下一寄存块号 |

| 步骤 | 块回收算法 |
|:----:|:---|
| 1 | 输入：待回收的块号 |
| 2 | 根据块号计算所属块组 |
| 3 | 若栈未满，将块号压入栈顶，空闲块数加一 |
| 4 | 若栈已满（已达 50 个）： |
| 5 | 　　将整个栈的 50 个块号和当前链表头写入待回收块的数据区 |
| 6 | 　　将待回收块设为新的链表头 |
| 7 | 　　清空栈，将待回收块号放入栈底，栈内计数置为 1 |
| 8 | 　　空闲块数加一 |

**复杂度**：分配和回收的均摊时间复杂度为 O(1)。每 50 次操作中最多需要一次额外的磁盘读写（栈补充或溢出写入），平均每次操作不到 1/50 次额外磁盘 I/O。

---

### 11.2 动态 inode 分块分配与 B+ 树索引

**设计思路**：传统 ext2 文件系统在格式化时预留固定的 inode 区域，导致 inode 数量在创建时已确定，无法灵活调整。本项目借鉴 XFS 的动态 inode 分配思想：不预留 inode 区，而是按需从普通数据块中分配 inode 块（每块存放 16 个 inode，称为一个"inode 块"）。为了快速定位任意 inode 所在的物理位置，系统维护两棵 B+ 树：

- **位置树**：以 inode 编号为键，叶节点存储 `(inode号, 块号, 槽位号)` 三元组，用于快速定位 inode 的磁盘位置。
- **空闲树**：以 inode 块号为键，叶节点存储 `(块号, 空闲位图)` 二元组，16 位位图标记每个槽位是否空闲。

此外，系统还维护一个容量为 128 的空闲 inode 号回收栈，回收的 inode 优先入栈，分配时优先从栈中取出，避免频繁的 B+ 树操作。

**伪代码**：

| 步骤 | inode 分配算法 |
|:----:|:---|
| 1 | 输入：父 inode 编号（用于局部性提示） |
| 2 | 尝试从空闲 inode 栈中弹出一个 inode 号 |
| 3 | **若栈非空**（命中快速路径）： |
| 4 | 　　通过位置树查找该 inode 的物理位置（块号和槽位） |
| 5 | 　　在空闲树中将对应槽位的空闲标记清除 |
| 6 | 　　将该 inode 的磁盘内容清零，返回 inode 号 |
| 7 | **若栈为空**（慢速路径）： |
| 8 | 　　查找父 inode 所在块组，记为目标块组 |
| 9 | 　　在空闲树中搜索目标块组内有空闲槽位的 inode 块 |
| 10 | 　　**若找到**：取出空闲槽位，清除空闲标记 |
| 11 | 　　**若未找到**：从目标块组分配一个新数据块作为 inode 块 |
| 12 | 　　　　初始化该块为全零 |
| 13 | 　　　　在空闲树中注册该块，位图为 16 位全 1（除第 0 槽外） |
| 14 | 　　　　取第 0 槽作为分配结果 |
| 15 | 　　生成新的全局唯一 inode 编号 |
| 16 | 　　在位置树中插入 `(inode号, 块号, 槽位号)` |
| 17 | 　　返回 inode 号 |

| 步骤 | 位置树叶节点分裂 |
|:----:|:---|
| 1 | 当叶节点已满（63 条记录），无法插入新记录 |
| 2 | 将叶节点的记录从中间一分为二 |
| 3 | 前半部分保留在原叶节点，后半部分写入新分配的叶节点 |
| 4 | **若当前为单层树**（无索引层）： |
| 5 | 　　分配一个新的索引节点作为新的根 |
| 6 | 　　索引节点包含两个子指针，分别指向原叶和新叶 |
| 7 | 　　树的层级从 0 升为 1 |
| 8 | **若已有索引层**： |
| 9 | 　　在父索引节点中插入指向新叶节点的条目 |
| 10 | 　　更新原条目的分界键 |
| 11 | 重新执行插入操作 |

| 步骤 | inode 回收算法 |
|:----:|:---|
| 1 | 输入：待回收的 inode 编号 |
| 2 | 在位置树中查找该 inode 的物理位置 |
| 3 | 将该位置的磁盘 inode 清零 |
| 4 | 在空闲树中，将对应槽位的空闲标记置为 1 |
| 5 | 将 inode 编号压入空闲 inode 栈（若未满） |
| 6 | 已用 inode 计数减一 |

**复杂度**：位置树查找为 O(log N)，其中 N 为 inode 总数。快速路径（栈命中）分配为 O(log N)（一次位置树查找 + 一次空闲树更新）。慢速路径（新块分配）额外增加 O(1) 的块分配和 B+ 树插入开销。单个叶节点最多容纳 63 条记录，两层 B+ 树即可管理 63×63 = 3969 个 inode，满足系统最大 8064 个 inode 的需求。

---

### 11.3 Extent 变长区段映射与自动合并

**设计思路**：传统的间接块索引（如 ext2 的一级/二级/三级间接块）对每个逻辑块单独记录物理块号，存储开销大且大文件定位慢。本项目采用类似 ext4 的 Extent 映射方案：每条 Extent 记录描述一段连续的逻辑块到物理块的对应关系，只需 8 字节即可表达数百个连续块的映射。

系统采用三级存储策略：
1. **内联 Extent**：inode 中直接存放一条 Extent（8 字节），对于只占用一段连续块的小文件，无需任何额外存储。
2. **单叶节点**：当文件不连续时，分配一个叶节点块（最多存放 62 条 Extent），可描述最多 62 段不连续区域。
3. **B+ 树**：当叶节点不够时，扩展为索引节点 + 多叶节点的 B+ 树结构。

**关键创新**——自动合并：每次插入新 Extent 后，系统自动检查相邻记录，如果逻辑地址和物理地址均连续，则合并为一条更长的 Extent。这使得顺序写入的文件即便经过多次写入操作，最终也只保留极少的 Extent 记录。

**伪代码**：

| 步骤 | Extent 块映射算法（查找或分配） |
|:----:|:---|
| 1 | 输入：内存 inode、逻辑块号、是否允许分配新块 |
| 2 | 先检查 inode 内联 Extent 是否覆盖该逻辑块 |
| 3 | 　　若命中，直接计算物理块号返回 |
| 4 | 若 inode 有 Extent 树，在树中查找对应叶节点 |
| 5 | 　　遍历叶节点中的 Extent 数组，查找覆盖该逻辑块的记录 |
| 6 | 　　若命中，计算并返回物理块号 |
| 7 | 若未命中且不允许分配，返回失败 |
| 8 | 分配一个新物理块 |
| 9 | 尝试**快速追加**：若新逻辑块号和新物理块号恰好接在内联 Extent 末尾 |
| 10 | 　　直接将内联 Extent 的长度加一，返回新物理块号 |
| 11 | 否则，构造新 Extent 记录并执行**插入 Extent** |

| 步骤 | 插入 Extent 算法 |
|:----:|:---|
| 1 | 输入：内存 inode、新 Extent 记录 |
| 2 | **情况一**：inode 无 Extent 树且内联 Extent 为空 → 直接写入内联 Extent |
| 3 | **情况二**：inode 无 Extent 树但内联已占用 → 合并尝试 |
| 4 | 　　将内联 Extent 和新 Extent 排序后尝试合并 |
| 5 | 　　若合并成功（两段物理连续），更新内联 Extent，完成 |
| 6 | 　　若无法合并，分配叶节点块，将两条记录写入 |
| 7 | 　　将叶节点块号记入 inode 的树根字段，清空内联 Extent |
| 8 | **情况三**：inode 已有 Extent 树 → 定位到正确的叶节点 |
| 9 | 　　在叶节点数组中按逻辑块号有序插入新 Extent |
| 10 | 　　执行**相邻合并**扫描 |

| 步骤 | 相邻合并算法 |
|:----:|:---|
| 1 | 输入：Extent 数组（已按逻辑块号排序）、数组长度 |
| 2 | 标记"已变化"为真 |
| 3 | **当"已变化"为真时循环**： |
| 4 | 　　将"已变化"置为假 |
| 5 | 　　对数组中相邻的每一对记录(i, i+1)： |
| 6 | 　　　　计算第 i 条记录的逻辑结束位置 = 起始逻辑块号 + 长度 |
| 7 | 　　　　计算第 i 条记录的物理结束位置 = 起始物理块号 + 长度 |
| 8 | 　　　　若逻辑结束 = 第 i+1 条起始逻辑块号 **且** 物理结束 = 第 i+1 条起始物理块号 |
| 9 | 　　　　　　将第 i 条的长度加上第 i+1 条的长度 |
| 10 | 　　　　　　删除第 i+1 条，数组长度减一 |
| 11 | 　　　　　　将"已变化"置为真，跳出内层循环重新扫描 |

**复杂度**：内联 Extent 的查找为 O(1)；B+ 树查找为 O(log N)，其中 N 为叶节点数。合并扫描为 O(K)，K 为叶节点中的 Extent 数（最多 62）。对于顺序写入的文件，快速追加路径使每次写入的映射更新为 O(1)。

### 11.4 元数据日志与崩溃恢复

**设计思路**：文件系统的元数据修改（如 inode、锚块、超级块的写入）涉及多个磁盘块的联动更新。如果在更新过程中发生崩溃（断电或进程异常终止），可能导致磁盘上的元数据处于不一致状态。本项目采用先写日志再回写原位的 WAL（Write-Ahead Logging）策略，确保崩溃后可以通过重放日志恢复到一致状态。

日志区占据磁盘的最后 32 个块，第一个块为日志头，其余 31 个块为日志数据槽。

**伪代码**：

| 步骤 | 事务提交算法 |
|:----:|:---|
| 1 | 读取日志头，将序列号加一 |
| 2 | 将本次事务的日志条目数写入日志头，提交标志置为"未完成"（0） |
| 3 | **写入日志头到磁盘**（此时磁盘上标记为有未完成事务） |
| 4 | **循环**：将每条日志记录（原始块号 + 块的完整副本）写入日志数据槽 |
| 5 | 全部日志写入成功后，将日志头的提交标志改为"已完成"（1） |
| 6 | **写入日志头到磁盘**（此时磁盘上标记事务已提交） |
| 7 | **循环**：将每条日志记录的数据副本回写到原始块号位置 |
| 8 | 同步更新块缓存中对应块的内容 |
| 9 | 清除事务状态 |

| 步骤 | 崩溃恢复（日志重放）算法 |
|:----:|:---|
| 1 | 挂载文件系统时，读取日志头 |
| 2 | 检查日志头魔数是否合法 |
| 3 | 若提交标志为"已完成"（1）**且**日志条目数大于 0： |
| 4 | 　　说明上次事务的日志写入已完成，但原位回写可能未完成 |
| 5 | 　　**循环**读取每条日志记录，将数据副本重新写入原始块号 |
| 6 | 　　重放完成后，将日志条目数归零，重写日志头 |
| 7 | 若提交标志为"未完成"（0）： |
| 8 | 　　说明上次事务的日志写入未完成，日志数据不可信 |
| 9 | 　　忽略此次日志，磁盘保持上一次成功提交时的状态 |

**安全性保证**：
- 崩溃发生在步骤 3 之前：磁盘无变化，安全。
- 崩溃发生在步骤 3~4 之间：日志头标记未完成，重放时忽略，安全。
- 崩溃发生在步骤 5~6 之间：日志数据已完整，但标记尚未改为已完成。日志头仍为未完成，重放时忽略。磁盘原位数据可能部分更新，但由于标记为已完成后才开始回写，此处不会出现部分回写（提交标志与回写是严格有序的）。
- 崩溃发生在步骤 7 之后：日志已提交且部分回写，重放时重新回写所有记录即可恢复一致性。

### 11.5 加盐迭代密码哈希

**设计思路**：简单的哈希函数容易被彩虹表攻击或暴力破解。本项目实现了一种自定义的加盐迭代哈希算法，将密码与随机盐值混合后经过大量轮次的位运算，产生 256 位的哈希值。

**伪代码**：

| 步骤 | 密码哈希生成算法 |
|:----:|:---|
| 1 | 输入：明文密码、16 字节随机盐值 |
| 2 | 初始化 8 个 32 位状态字，使用固定的种子值填充 |
| 3 | 将盐值与密码拼接为一个混合数据序列 |
| 4 | **执行 10000 轮迭代**，每轮： |
| 5 | 　　从混合序列中循环取出一个字节 B |
| 6 | 　　根据 B 的低 3 位选择一个状态字（共 8 个） |
| 7 | 　　将该状态字与 B 做异或 |
| 8 | 　　将该状态字与另一个状态字（循环偏移 7 位）做异或 |
| 9 | 　　将该状态字循环左移 (B mod 32) 位 |
| 10 | 迭代结束后，8 个状态字依次输出为 32 字节哈希值 |

**安全性特点**：
- **加盐**：每个用户的盐值不同，相同密码产生不同的哈希，防止彩虹表攻击。
- **高迭代**：10000 轮运算使单次验证耗时增加，显著提高暴力破解的成本。
- **非线性混合**：异或、循环移位和跨字混合三种运算组合，增强了雪崩效应（输入微小变化导致输出大幅改变）。

### 11.6 两遍汇编器的标签解析

**设计思路**：汇编源代码中的标签（如跳转目标、数据地址）在引用时可能出现在定义之前（前向引用），单遍扫描无法确定标签的确切地址。两遍扫描方案在第一遍收集所有标签的地址，第二遍根据已知地址生成最终的机器码。

**伪代码**：

| 步骤 | 第一遍扫描（收集标签） |
|:----:|:---|
| 1 | 初始化：当前段 = 代码段，代码偏移 = 0，数据偏移 = 0 |
| 2 | **逐行扫描**源文件： |
| 3 | 　　跳过空行和注释行 |
| 4 | 　　若行末有冒号，识别为标签定义 |
| 5 | 　　　　记录标签名 → 当前段偏移的映射 |
| 6 | 　　若为段切换伪指令（如 .text、.data、.bss）： |
| 7 | 　　　　切换当前段 |
| 8 | 　　若为数据伪指令（如 .word、.space、.ascii）： |
| 9 | 　　　　根据数据长度推进数据偏移 |
| 10 | 　　若为普通指令： |
| 11 | 　　　　代码偏移加 4（每条指令 4 字节） |
| 12 | 　　若为栈大小伪指令（.stack）： |
| 13 | 　　　　记录请求的栈大小 |
| 14 | 第一遍结束，所有标签地址已确定 |

| 步骤 | 第二遍扫描（生成机器码） |
|:----:|:---|
| 1 | 初始化：重置偏移，分配代码缓冲区和数据缓冲区 |
| 2 | **逐行扫描**源文件（与第一遍相同的分词流程）： |
| 3 | 　　跳过标签行、空行和注释行 |
| 4 | 　　若为数据伪指令，将数据写入数据缓冲区对应偏移 |
| 5 | 　　若为普通指令： |
| 6 | 　　　　解析助记符，确定操作码 |
| 7 | 　　　　解析操作数（寄存器号、立即数或标签引用） |
| 8 | 　　　　若操作数为标签名，从第一遍的映射表中查找地址 |
| 9 | 　　　　对跳转类指令，计算标签地址相对当前指令的偏移量 |
| 10 | 　　　　按 32 位编码格式组装指令字 |
| 11 | 　　　　写入代码缓冲区 |
| 12 | 组装 UPX 文件头（魔数、入口点、各段大小） |
| 13 | 将文件头 + 代码段 + 数据段拼接为完整的二进制文件 |

**复杂度**：两遍扫描均为 O(L)，L 为源文件行数。标签查找使用线性扫描（标签数通常极少），总体时间与源文件大小成正比。
